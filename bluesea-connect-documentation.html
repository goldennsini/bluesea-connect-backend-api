1. Introduction & Architecture Overview

BlueSea Connect is a financial gateway infrastructure that enables users to deposit from BlueSea wallets into partner platforms, verify partner accounts, process transfers, and receive withdrawals. It is designed as a provider integration engine similar to Stripe Connect, Paystack, and Flutterwave.

1.1 Core Architecture Principle

The frontend NEVER communicates directly with partner APIs. All provider interactions flow through the BlueSea backend:

Request Flow:

Frontend (React) → BlueSea API (FastAPI) → Provider Adapter → Partner API

5.2 Wallet Balance Constraints

CHECK (available_balance >= 0)
CHECK (locked_balance >= 0)
CHECK (total_balance = available_balance + locked_balance)

5.3 Transaction Status Workflow

Transactions flow through these states:

PENDING -> PROCESSING -> SUCCESS
                      -> FAILED (retryable)
                      -> REVERSED (from success)
Critical Safety Rule
Wallet funds are NEVER permanently deducted before provider confirmation. Funds are locked during processing and only deducted upon success, or released upon failure.

8. Transaction Flow

8.1 Deposit Flow (Outgoing)

  1. Validate Request - Check partner exists, supports deposits, amount within limits
  2. Authenticate - Validate JWT token
  3. Verify PIN - Check transaction PIN against bcrypt hash
  4. Check Balance - Verify sufficient available balance (including fees)
  5. Generate Reference - Create unique transaction reference (BSC-XXXXXXXX-XXXX)
  6. Create Transaction - Insert PENDING record
  7. Lock Funds - Move amount from available to locked (atomic UPDATE)
  8. Update Status - Set PROCESSING
  9. Call Provider - Send payment through provider adapter
  10. Process Response
    • Success: Complete locked funds (deduct from locked + total), set SUCCESS
    • Processing: Keep locked, background polling continues
    • Failure: Release locked funds back to available, set FAILED
  11. Return Response - Frontend-safe normalized response

8.2 Withdrawal Flow (Incoming)

  1. Validate request (partner, amount limits)
  2. Authenticate and verify PIN
  3. Create PENDING transaction (no fund lock - incoming)
  4. Call provider to initiate withdrawal
  5. On success: Credit wallet with net amount (after fees)
  6. Return response with estimated arrival time

9. Webhook System

9.1 Supported Events

Event TypeDescription

payment.failedDeposit failed
withdrawal.successWithdrawal completed
withdrawal.failedWithdrawal failed
transaction.reversedTransaction was reversed

9.2 Webhook Delivery

  • Signature - HMAC-SHA256 with timestamp + payload
  • Retry Policy - 5 attempts with exponential backoff: 1min, 5min, 15min, 30min, 1hr
  • Replay Protection - Timestamp tolerance window (5 minutes)
  • Delivery Log - All attempts logged with response status and body

13. Security Considerations

13.1 Implemented Security Measures

  • JWT Authentication - Short-lived access tokens (15 min) with refresh tokens
  • Transaction PIN - Separate 4-6 digit PIN, bcrypt hashed
  • HMAC Signatures - All provider requests signed, webhooks verified
  • Rate Limiting - Redis-based sliding window (100 req/min default)
  • Replay Protection - Webhook timestamp tolerance prevents replay attacks
  • API Key Security - SHA-256 hashed secrets, never stored plaintext
  • IP Whitelist - Per-API-key IP restrictions
  • Audit Logging - All actions logged with before/after state
  • Fund Safety - Atomic operations, never deduct before confirmation
  • Distributed Locks - Prevent double-spending race conditions

13.2 Security Best Practices

Production Checklist
  • Change all default secrets in .env
  • Enable HTTPS only (no HTTP in production)
  • Restrict CORS origins (don't use "*")
  • Run database migrations before starting
  • Monitor Celery worker health
  • Enable database backups
  • Set up log aggregation

Summary

BlueSea Connect is a production-ready fintech gateway built for scalability, security, and reliability. The architecture ensures:

  • Frontend isolation from provider APIs
  • Safe transaction processing with fund locking
  • Comprehensive audit trail for all operations
  • Extensible provider system for adding new partners
  • Reliable webhook delivery with retry logic
  • Background job processing for async operations

The codebase is organized into clear modules following the dependency inversion principle - the core business logic (services) is independent of HTTP transport (routes) and external providers (adapters).
